# Typed-response rollout — backend OpenAPI spec gaps

## Relationship to the backend's `proposal/typed-response-dtos` branch

This catalog is the SDK-side bridge to the backend workstream tracked
on `openbox-backend`'s `proposal/typed-response-dtos` branch. That
proposal annotates NestJS controllers with `@ApiOkResponse({ type:
... })` so the autogenerated OpenAPI document gets schemas; the SDK's
codegen then picks them up automatically. The Agent module is a
starter on that branch; the other modules are pending.

This file lists the same endpoints the backend proposal needs to
cover, but as an SDK-side punch list with the typed-model shape we
should declare in our local `specs/typespec/backend/responses.tsp`
until the backend lands `@ApiOkResponse`. The two paths converge:
once an endpoint is annotated upstream, the corresponding entry in
`responses.tsp` (or the matching `: OkResponse` op) gets dropped and
consumers transparently start using the auto-generated schema. The
mechanism is the one the existing `responses.tsp` doc-comment
already describes; this catalog enumerates the work.



44 backend endpoints currently emit `responses: { 200: { description: "" } }`
with no schema. The SDK's `ResponseOf<>` extractor (just fixed in
codegen/emitters/typespec-emitter-typescript) now correctly looks up
the 200 schema, but for these endpoints there is nothing to look up —
they bottom out at `unknown`. Each one is a typed-API gap on the SDK
side.

How the gap surfaces: `await client.<method>(...)` returns
`unknown`, every consumer either casts at the call site (visible drift)
or uses a hand-written wrapper (legacy, just removed). To restore typed
returns, each op below needs `: OkResponse` replaced with a typed
model declared in `specs/typespec/backend/responses.tsp` (or a
sub-file). Where the model already exists, point at it; otherwise
design a new one against the actual NestJS controller response shape
under `src/modules/<area>/<area>.controller.ts` in the backend.

## Rollout

For each endpoint:

1. Read the NestJS controller method (in
   the backend's `src/modules/<area>/` controllers and services).
2. Read the service method it delegates to to confirm the literal
   return shape.
3. Add a typed model to `specs/typespec/backend/responses.tsp` (or a
   per-area sub-file) matching that shape.
4. Replace `: OkResponse` with `: Body<NewModel>` (or just `: NewModel`
   when wrapped via `@useAuth`) on the op.
5. Regenerate the spec via `npm run specs:all` and confirm
   `client.<method>()` resolves to the typed shape, not `unknown`.
6. Drop any cast workarounds at call sites (e.g. polling/index.ts's
   getOrgApprovals cast that's no longer needed once that endpoint is
   already typed).

## Why this is filed as a catalog rather than one big PR

Each endpoint's response shape is determined by the backend's actual
return value; designing the model needs reading the controller. Doing
44 of those in one branch is an audit-and-design exercise, not a
mechanical rewrite. Per-batch rollout (e.g. metrics in one branch,
dashboards in another) keeps each diff reviewable and lets the spec
drift-detection on the typed-responses-rollout proposal land
incrementally.

The fix that makes this rollout matter (the `ResponseOf<>` Record-of-
both-200-and-201 bug) already shipped on the cursor-plugin tree;
without it every typed response was being silently downgraded to
`unknown` regardless of whether the spec carried a schema.
